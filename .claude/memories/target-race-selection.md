# 対象レース選定の工程（週次）

> **状態**: 方針メモ（ドラフト）。いずれ `run-weekly` の前段ワークフロー（pack 作成の手前）の1ステップへ昇格する。
> **確定**: 2026-06-20（Sho 合意）。
> **根拠**: `docs/worldbuilding/yosouya-10/円卓制度.md`（「円卓に座る者は、毎週1レースの予想を出す」「G1戦」）、`docs/design/weekly-ops.md`（週次 runbook・race_id 形式・§3.5 タイミング）。

## 方針（一行）

**毎週1本、その週末の JRA 中央開催の最上位重賞（G1優先 → G2 → G3）を対象にする。**

## 選定ルール

1. **スコープ**: JRA 中央開催のみ。地方競馬（南関東等）は対象外。
   理由: pack・オッズ・スクレイピングの前提が JRA に揃っているため。地方を入れるならデータ源整備が別途必要（→ 将来課題）。
2. **主基準（格）**: その週末（土＋日）の重賞のうち、**最上位の格を1本**。G1 > G2 > G3。
   円卓制度の「毎週1レース」＋「G1戦は特別扱い（重い評価・外部挑戦者増）」に合致。データ密度・市場流動性も最上位重賞が最も厚い。初回ライブ運用（2026 宝塚記念）もこれに該当。
3. **本数**: 原則 **1本**（canon）。ただし **G1連戦など特別な週は複数可**。
   複数を対象にする場合は、各レースに独立した `runs/<race_id>/` を作って並走させる。
4. **タイブレーク（暫定）**: 最上位の格が複数並ぶ週は、**市場流動性（売上）が大きい方**を採る。迷えば Sho 判断。
   ※ Sho 未確定のため暫定。「基本は週1回・最大格だけ」が方針なので発火頻度は低い想定。実運用で頻発するようなら見直す。
5. **除外**: 出馬表が未確定のレースは選ばない（pack が組めない）。pack 作成は **T-1日夜**が基準（§3.5）なので、それまでに出走馬・枠順が確定する重賞を選ぶ。

## 手順（決定 → 着手）

1. その週末の **JRA 重賞を列挙**（JRA 重賞日程 / netkeiba 重賞カレンダー）。
2. 上記ルール 2〜4 で **最上位の格を1本**選ぶ。
3. **race_id を確定**。netkeiba 12桁形式 = `西暦4 + 場2 + 回2 + 日2 + R2`（例: 2026 宝塚記念 = `202609030411`、阪神 = 場 `09`）。
4. `runs/<race_id>/` を作成 → **pack-v1.json 作成**へ（`run-weekly` のステップ1ハードゲート）。以降は `scripts/run-weekly.sh <race_id>` に接続。
5. **タイミング**: pack v1 は T-1日夜。当日朝にオッズ依存組のみ pack v2 で再取得・再実行可（§3.5 / weekly-ops フェーズ1）。

# 競馬予想AIハーネス — 10人の専門家subagent設計書 作成プラン

## Context

- リポジトリ `k-ba-man` はほぼ空（`.vscode` のみ）。ここに競馬予想AIハーネスを作っていく。
- 戦略は**集合知**: 日本のファーストネームを持つ10人の「競馬予想専門家」subagent が、各自異なる流派・バックエンド（gemini CLI / codex CLI / Haiku / Sonnet / Opus）・effort で順位予想 → 集合知理論で集約（将来）→ 期待値ベースの資金配分 skill（将来）。
- **当面の目標レース: 2026年 宝塚記念（芝2200m）に全力投入**。ハーネス構造は汎用に保ちつつ、専門家の知識設計・プロンプト骨子はこの一戦に最適化する。
- 今回のスコープ: **10 subagent の設計書 1枚（md）** ＋ リサーチメモ（docs/）。エージェント実装・集約 skill・資金配分 skill は対象外（ただし設計書はそれらとの接続＝出力スキーマ契約を見据える）。
- トークン効率: web リサーチとメモ執筆は Haiku サブエージェントに委譲。設計の統合・執筆のみ私（メインループ）が行う。
- 設計書の粒度: **仕様＋プロンプト骨子**（全文化は実装フェーズ）。

## 実行ステップ

### Step 0: 準備
- 本プランを `./.claude/plans/keiba-10-experts-design.md` にも複製（プラン両置きルール。リポジトリは空なので命名はこれが基準になる）
- プロジェクトメモリに「k-ba-man = 競馬予想集合知ハーネス、目標: 宝塚記念2026」を記録

### Step 1: リサーチ委譲（Haiku ×4 並列、各自がメモ執筆まで担当）
| # | 担当 | 内容 | 出力 |
|---|---|---|---|
| R1 | Haiku/web | 宝塚記念2026: 開催日・開催場・コース形態（阪神芝2200m内回りか要確認）、出走予定馬・ファン投票動向、過去10年の傾向（リピーター・前走ローテ・枠・馬場） | takarazuka-kinen-2026.md |
| R2 | Haiku/web | 日本競馬の主要予想流派とその手法・データソース: 血統/スピード指数/展開/調教/騎手・厩舎/オッズ・市場/馬場・トラックバイアス/穴狙い・本命戦略 | keiba-prediction-schools.md |

- 各エージェントは「メモを docs/research/ に書き、5行以内の要約だけ私に返す」指示でトークン節約
- 注意: 阪神競馬場改修の影響で近年の宝塚記念は開催場が変動している（2024年は京都）。2026年の開催場・コースは必ず web で確認し、推測で書かない

### Step 2: 設計書執筆（私 = メインループ）
リサーチ結果を統合し `docs/design/expert-subagents.md` を執筆。構成:

1. **コンセプトと理論的根拠** — 多様性予測定理: 集団誤差 = 平均個人誤差 − 多様性。流派×モデル×effort を散らすことが理論的に正当である、という柱
2. **全体アーキテクチャ** — オーケストレーター →（レースデータパック配布）→ 10人並列予想 → 集約 skill（将来）→ 資金配分 skill（将来）。後段2つはインターフェースのみ定義
3. **共通仕様**
   - 入力: レースデータパック（出馬表・オッズ・馬場・過去走等をオーケストレーターが一括取得し全員に同一配布。各自の追加調査は流派次第で許可）
   - 出力スキーマ（全員共通JSON）: `predicted_ranking`（着順予想・馬番）、`win_prob`（勝率分布）、`confidence`、印（◎○▲△）、短い根拠 — 将来の集約 skill が読む契約
   - **独立性原則**: 他の専門家の予想は見せない（herding 防止。集合知の前提条件）
   - 運用制約: タイムアウト、CLI 不在時の fallback、外部アクセスは読み取りのみ、馬券購入はしない（予想と配分計算まで）
4. **10人のプロフィール**（各人: 名前・人物像・流派・バックエンド/モデル・effort・プロンプト骨子・宝塚記念2200mでの着眼点・データ要求）
5. **多様性マトリクス** — 流派×バックエンド×effort の重複チェック表
6. **実装ガイド** — `.claude/agents/*.md` 雛形（frontmatter）、gemini/codex はラッパースクリプト経由で呼ぶ方針（セキュリティルール準拠、スクリプトはコロケーション）
7. **将来スコープとの接続** — 集約 skill、資金配分 skill（Kelly 基準を想定した入力要件）、バックテスト構想

### 10人の編成ドラフト（設計書で精緻化。名前は仮）
| 名前(仮) | 流派 | バックエンド | effort |
|---|---|---|---|
| 龍之介 | 血統・配合（スタミナ/道悪適性血統） | Opus | 高 |
| 誠 | データ・統計（宝塚記念の歴史傾向） | codex CLI (reasoning=high) | 高 |
| 美咲 | 展開・ペース読み（2200m想定ラップ） | Opus | 中 |
| 健太 | スピード指数 | Sonnet | 中 |
| 鉄平 | 調教・仕上がり | codex CLI (default) | 中 |
| さくら | オッズ・市場分析（ファン投票×オッズ乖離） | Haiku | 低 |
| 葵 | 騎手・厩舎 | Sonnet | 中 |
| 陽菜 | 穴党・逆張り | gemini CLI (flash) | 低 |
| 優子 | 堅実本命派 | Haiku | 低 |
| 吾郎 | 馬場読み・トラックバイアス（梅雨の道悪） | gemini CLI (pro) | 高 |

- 配分: gemini×2 / codex×2 / Haiku×2 / Sonnet×2 / Opus×2、effort 高3:中4:低3
- 低effort組も「即断即決を信条とするプロ」として人物造形し、専門家であることは崩さない
- effort の実現方法はバックエンド毎に異なる: Claude系=プロンプトでの思考深度指示、codex=reasoning effort 設定、gemini=pro/flash のモデル選択（R4 の調査結果で確定）
- 名前は師匠（Sho）と被らないものにする

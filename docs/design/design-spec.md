# k-ba-man サイト デザイン要件定義書 v2

## 1. プロダクト概要

k-ba-man は、10人の AI 予想専門家（キャラクター）が毎週の重賞レースを予想し、その過程・結果・ランキング変動をコンテンツとして公開するメディアサイト。

### 1-1. サイトが果たすべき役割

- 10人のキャラクターを「見たい・応援したい」と思わせる入口になる
- 予想の過程と結果を、エンタメとして消費できる形で届ける
- レースごとの成績変動が「物語の続き」として機能する
- 初見のユーザーに「何をやっているサイトか」を5秒で伝える
- 「この席に座るのは怖い」「でも降りられない」という緊張感が伝わる
- 外した週に何が起きるか（流派の存在意義への問い）が構造として見える
- 「次週どうなるか」が気になって再訪する理由がある

### 1-2. 想定ユーザー

| セグメント | 来訪動機 | 重視するもの |
|---|---|---|
| 競馬ファン | 予想の中身を見たい | 買い目・根拠・的中率 |
| キャラクターファン | キャラの物語を追いたい | ビジュアル・個性・関係性 |
| テック層 | AI×集合知の仕組みに興味 | アーキテクチャ・スコアリング |
| 初見の通りすがり | SNSで流れてきた | 世界観・ビジュアルのインパクト |

## 2. 情報アーキテクチャ

### 2-1. ページ構成（4ページ + モーダル）

ナビ項目は4つに統合し、モバイルでの認知負荷とタブバーの収まりを最適化する。

| ページ | パス | 核心的な問い |
|---|---|---|
| Overview | `#/overview` | 「k-ba-man とは何か」「いま誰が強いか」 |
| 予想ライブ | `#/live` | 「今週の予想はどうなっているか」 |
| 成績 | `#/results` | 「10人の序列と、レースごとの成績」 |
| キャラクター一覧 | `#/characters` | 「誰がいるか・どんな流派か」 |
| キャラクター詳細 | `#/characters/:id` | 「この人はどういう予想屋か」 |
| レース詳細モーダル | (dialog) | 「このレースの印・買い目・払戻」 |

**統合の判断**: 旧「ランキング」と「成績履歴」は同一データの2つの切り口（縦軸=キャラ / 横軸=レース）。「成績」ページ内にタブ切り替え（ランキング表示 / レース別表示）を設ける。

**About 情報**: テック層向けの仕組み解説は、初期は Overview のフッター付近に折りたたみセクションとして配置。予想パイプライン（10人予想→集約→配分→買い目）の概要、Brier スコアの解説、集合知の設計思想を含む。

### 2-2. 各ページの情報要素

#### Overview
- ヒーロー：プロジェクトのアイデンティティを一枚で示すビジュアル
- **予想ライブバナー**（リピーター向け）：今週の対象レース名 + フェーズ状態 + 直リンク。ライブデータがある場合のみ表示。ヒーロー直下に配置し、リピーターはスクロールせずに目的地へ到達できる
- **「今週のドラマ」帯**: 直近レース結果から導かれるナラティブ1〜2文を動的に表示。席次変動の最大上昇/下降キャラ、集合知◎の的中/外れ、単独的中した少数派キャラなど
- サマリー統計：記録レース数、累計収支、的中率、暫定首位
- 現在の上位3名：ミニランクカード
- キャラクター一覧への導線
- **最新レース結果のダイジェスト**: 直近1レースの結果（勝ち馬・収支・ランク変動ハイライト）
- About セクション（折りたたみ）：仕組みの概要

#### 予想ライブ（3フェーズ制）
- フェーズ: 告知(announce) → 予想公開(predictions) → 結果(result)
- レース情報：名称・発走・条件・状態
- 集合知の予想：◎○▲△の印、上位馬の確率分布
- 10人の個別予想：各キャラの印・**根拠テキスト**・自信度
- **対立の可視化**: 集合知と大きくずれた個別予想にマーカーを付ける（集合知◎と個人◎が異なるときの視覚的強調）
- 結果フェーズ：着順・払戻・各キャラの答え合わせ・ランク変動
- 結果フェーズ追加：各キャラの**直近N戦の◎結果をインライン表示**（例：「直近3戦 ◎1着→◎圏外→◎3着内」）

#### 成績（ランキング + レース別のタブ切り替え）
- **ランキングタブ**:
  - ソート可能な一覧（暫定席次・Brier・本命実績・網羅平均・LogLoss）
  - 各行：順位、キャラ画像・名前・二つ名、主要指標×5列
  - **降格圏の視覚表現**: 暫定スコアの低い者に「席が危うい」印象を与える演出
  - スコアリングの解説セクション
- **レース別タブ**:
  - 累計統計：投資・払戻・収支・回収率・的中率
  - レースカード×N：レース名・日付・条件・結果・収支
  - レース詳細モーダル：印、買い目一覧、払戻一覧、勝ち筋の解説

#### キャラクター一覧
- 10人分のカード
- 各カード：立ち絵、席番号、名前、二つ名、直近成績サマリー

#### キャラクター詳細
- プロフィールヒーロー：大きな立ち絵 + 背景
- 基本情報：名前、二つ名、流派、座右の銘
- 成績サマリー：暫定席次、平均 Brier、◎3着内率、最新◎
- **「この席に残る理由」セクション**: drama テキストを独立した視覚ブロックとして扱う
- 紹介文：この予想屋が何を見るか
- 強み・弱点
- **関連キャラクター**: 対立・補完関係にあるキャラ（例：鉄平の弱点「長期傾向や市場の説明」→さくら・誠）。各キャラの weakness テキストに含まれる他キャラ名から導出
- 成績履歴：レースごとの印と結果

### 2-3. ページ間導線マトリクス

| 遷移元 → 遷移先 | トリガー | 優先度 |
|---|---|---|
| Overview → 予想ライブ | ヒーロー CTA / ライブバナー | **最高** |
| Overview → 成績 | ヒーロー CTA / 上位3名カード | 高 |
| Overview → キャラクター一覧 | ヒーロー CTA / キャラグリッド | 高 |
| Overview → キャラクター詳細 | 上位3名カード / キャラカード | 中 |
| 予想ライブ → キャラクター詳細 | 個別予想カードのキャラ名・画像 | **最高** |
| 予想ライブ → 成績 | 結果フェーズの「詳細を見る」 | 中 |
| 成績 → キャラクター詳細 | ランキング行のキャラ名・画像 | **最高** |
| キャラクター一覧 → キャラクター詳細 | カードクリック | **最高** |
| キャラクター詳細 → キャラクター詳細 | 前後ナビ / 関連キャラリンク | 高 |
| キャラクター詳細 → 成績 | 成績セクションのリンク | 中 |
| 成績 → レース詳細モーダル | レースカードの「詳細を見る」 | 高 |
| **全ページ → 予想ライブ** | **グローバルナビ（ライブ中は強調表示）** | **最高** |

**設計原則**: 予想ライブが「今週のエントリポイント」。キャラクター詳細が「ハブ」として最多の導線を持つ。

## 3. コンテンツ素材

### 3-1. キャラクターデータ（10人）

各キャラクターが持つ属性:
- id, 席番号(01-10), 名前, かな, 流派名, 二つ名
- 一人称, 座右の銘（セリフ）
- テーマカラー（各キャラ固有）
- 紹介文（intro）, ドラマ（drama）
- 強み（3項目）, 弱点
- **対立・補完キャラ**（相関図から）: 例 龍之介⇔誠, 優子⇔陽菜
- **象徴小物**: 龍之介=巻物, 誠=タブレット, 美咲=指し棒, 健太=指数カード, 鉄平=ストップウォッチ, さくら=スマホ, 葵=双眼鏡, 陽菜=馬券, 優子=チェックボード, 吾郎=風速計

10人の流派:
1. 龍之介 — 血統・配合
2. 誠 — データ・統計
3. 美咲 — 展開・ペース読み
4. 健太 — スピード指数
5. 鉄平 — 調教・仕上がり
6. さくら — オッズ・市場分析
7. 葵 — 騎手・厩舎
8. 陽菜 — 穴党・逆張り
9. 優子 — 堅実本命
10. 吾郎 — 馬場読み・トラックバイアス

### 3-2. 画像アセット

- キャラクター立ち絵: `mini.png`(SD/チビ), `real.png`(実頭身) ×10人
- 集合ビジュアル: 10人集合の一枚絵
- 背景画像: 円卓の間、監査室、トラック朝焼け、マーケットブース（4種）
- ヒーローアート: アンサンブル戦闘シーン

#### 立ち絵の表示ルール

| コンテキスト | バリアント | 最小表示幅 | 切り抜き | 背景との関係 |
|---|---|---|---|---|
| キャラカード一覧 | `real.png` | 150px | バスト〜ウエスト中心 | キャラカラーのグラデーション背景 |
| ランキング行・ミニカード | `mini.png` | 44px | 円形マスク（顔中心） | キャラカラーのボーダー |
| プロフィールヒーロー | `real.png` | 300px | 全身。足元まで見せる | 背景アートの上にドロップシャドウで浮かせる |
| 予想カード（Live） | `mini.png` | 48px | 円形マスク | カード内、ヘッダー左寄せ |

- `real.png` を 150px 未満で使わない（小さい用途は必ず `mini.png`）
- キャラカードの画像比率は縦長を許容（最低 aspect-ratio: 3/4、推奨 2/3）
- 立ち絵の周囲に最低 8px の余白を確保（画像端がカード端に接しない）

### 3-3. 数値データ

- レース履歴 JSON（race-history.v1）
- ライブレース JSON（live-race.v1）
- 各種算出指標: Brier score, LogLoss, 的中率, 網羅率, 暫定スコア

## 4. デザイン要件

### 4-1. ブランド・トーン

- **席を守る切迫**: 10人がただの解説者ではなく、成績で席を失いうる競技者。「楽しいキャラ紹介サイト」ではなく「毎週裁かれる者たちの記録」
- **キャラクターが主役**: UI の装飾ではなく、ページの中心にキャラがいること
- **データとドラマの共存**: 数値は正確に読め、かつ物語の文脈に組み込まれていること
- **外れの重み**: 外した予想が消えない。負けを隠さない
- **集合知と個の矛盾**: 個人は席を守りたいが、集団には多様性が必要。この矛盾がサイト全体のテンションになる
- **週次の更新性**: レースごとにランキングが変わる動的な構造が伝わること

### 4-2. ビジュアル方向性の要件

#### ムード・世界観キーワード
- **「円卓」と「競馬場」の二重空間**: 室内の知的緊張感（暗い木と革の質感、卓上の書類）と屋外の躍動感（朝靄の芝、ゲートの金属感）
- **参照ジャンル**: スポーツ×群像ドラマ（『ブルーロック』の個対個の緊張感、『キングダム』の一覧性と勢力図、『賭ケグルイ』のデータと狂気の同居）
- **質感のキーワード**: 和紙のテクスチャ（明朝見出しとの親和性）、金属のヘアライン（精密機器のデータ感）、夜の競馬場の照明（温かい金系グロー × 冷たい環境光）
- **避けるべきもの**: ネオン/サイバーパンク、純粋なフラットUI、過剰なグラデーション

#### カラーシステム
- ダークテーマを基本（キャラ立ち絵が映えるため）
- **背景色はニュートラルな暗色**（純黒ではなく、微かに色味を帯びた暗色）。ページの性格によって色温度を変えてもよい
- 各キャラクターの固有カラーをアクセントとして活用（ボーダー、グロー、バッジ、背景のティント）
- **アクセントカラーの展開ルール**: 各キャラカラーから dim(10%)・mid(25%)・vivid(100%) の3段階を `color-mix()` 等で生成し、用途を統一
- **セマンティックカラー**: 的中(gold系)・3着内(紫/藍系)・外れ(赤系)・上昇(緑系)・下降(橙系) は全ページで統一
- 10人並べたとき「全員違う色」が一目で分かること

#### サーフェスとデプス
- 要素の階層を **surface-0（最奥の背景）→ surface-1 → surface-2 → surface-3（最前面カード）** の4段階以内で管理
- 浮いて見える要素には控えめなシャドウ（色付きグローよりドロップシャドウ寄り）
- データテーブル/統計は情報密度を保ちつつ読みやすいこと
- ヒーローセクションで「世界観」を一枚で伝えること

#### 流派の識別
- カラーに加え、各キャラに1つの**象徴アイコン/モチーフ**を定義し、カードやランキング行で使う
- 10色ストライプ（ゲートストライプ）はブランド要素として保持。ヒーロー下部やフッター上部に1箇所のみ使用

### 4-3. タイポグラフィ要件

- 日本語本文: 可読性の高いゴシック体
- 見出し/キャラ名: 世界観を伝える書体（明朝系 or 装飾系）
- 数値/指標: 等幅フォント（tabular nums）
- ラベル/メタ情報: 小さめ・控えめ

### 4-4. レイアウト要件

#### グリッドと最大幅
- コンテンツ最大幅: 1200px
- ページ左右パディング: `clamp(16px, 4vw, 40px)`
- レスポンシブ: 320px〜1920px を想定
- ブレイクポイント: モバイル(〜620px) / タブレット(〜880px) / デスクトップ(880px〜)
- ヒーロー・プロフィールヒーローはフルブリード（横幅いっぱい）
- コンテンツ領域はセンター寄せ

#### 余白のリズム
- **セクション間**: `clamp(48px, 8vw, 96px)`
- **セクション内ブロック間**: `clamp(24px, 4vw, 40px)`
- **カード/行の間隔（ガター）**: 8〜12px で統一
- **カード内パディング**: 14〜20px
- 余白は「大・中・小」の3段階を意識し、4段階以上に増やさない

#### カードグリッドのカラム数

| コンテキスト | デスクトップ | タブレット | モバイル |
|---|---|---|---|
| キャラクター一覧 | 5列 | 3列 | 1列 |
| 予想カード（Live） | 2列 | 2列 | 1列 |
| 統計タイル | 4列 | 2列 | 2列 |
| ランキング行 | 1行に全指標 | 主要指標のみ | スタック |

#### モバイルナビゲーション
- 620px 以下: **底部タブバー（固定）** を採用
  - 4ページの導線をアイコン＋短ラベルで常時表示
  - ハンバーガーメニューは不採用（4項目は収まる数であり、常時視認のメリットが大きい）
  - 底部バーの高さは 56〜64px、safe-area-inset-bottom を考慮
  - 現在地のハイライト（aria-current + ビジュアル強調）
- 620px〜880px: 横スクロール可能なタブバー（ヘッダー内）
- 880px〜: ヘッダー内水平ナビゲーション
- main 領域の下マージンを底部バーの高さ分確保

### 4-5. インタラクション要件

#### 基本操作
- ページ遷移: ハッシュルーティング（SPA）
- ソート: ランキングの並び替え
- タブ切り替え: 成績ページのランキング/レース別表示
- モーダル: レース詳細の展開
- アコーディオン: 成績履歴の開閉
- ホバー: カード・リンクのフィードバック
- reduced-motion 対応

#### モーション原則
- **基本トランジション**: 150ms ease — ホバー、ボタン押下、タブ切り替え
- **展開系トランジション**: 250〜350ms ease-out — アコーディオン、モーダル出現
- **ページ遷移**: コンテンツ差し替え後の scroll-to-top + fade-in（100〜150ms）
- **控えめが基本**: 要素が「飛ぶ」「回転する」「バウンスする」動きは使わない。translateY(-2〜-4px) 程度のリフト
- **カード一覧**: stagger（0.03s × index）でカードが順に現れる演出は可。total 0.3s 以内
- **prefers-reduced-motion** では即時切り替え

#### ホバー・フォーカスのフィードバック
- カード: わずかなリフト + ボーダーカラーの変化
- テキストリンク: 下線の色変化
- ボタン: 背景色のシフト + わずかなリフト
- ランキング行: 背景色のシフトのみ（表は静的に見えるべき）

#### タッチ操作
- タップ時に `:active` 状態で視覚フィードバック。ホバーの translateY はタッチデバイスでは `:active` に切り替え
- スワイプジェスチャは不採用（SPA のハッシュ遷移と競合）
- pull-to-refresh は不採用（更新頻度が週次）

#### フェーズ遷移の通知
- 静的SPAのため自動更新は不可。「最終更新日時」の表示と「データ更新はページリロードで反映」を明示
- 空状態の表示要件を定める（ライブにレース未登録時、成績0件時）

### 4-6. アクセシビリティ要件

- スキップリンク
- aria-label / aria-current で意味を補完
- キーボードナビゲーション（focus-visible）
- コントラスト比 WCAG AA 以上
- 画像に alt テキスト

### 4-7. パフォーマンス要件

#### 基本方針
- 外部フレームワーク不使用（vanilla HTML/CSS/JS）
- CSS カスタムプロパティによるテーマ管理

#### 画像最適化
- キャラクター画像に `width` / `height` 属性を明示し、CLS を防止
- 一覧ページ: mini.png を使用（目安: 1枚 30KB 以下）
- 詳細ページ: real.png は above-the-fold の1枚のみ `loading="eager" fetchpriority="high"`、他は lazy
- ヒーロー背景画像: `<link rel="preload" as="image">` でプリロード
- 将来的な WebP/AVIF 対応余地を残す

#### CLS 対策
- すべての画像コンテナに `aspect-ratio` or 固定高さを設定
- Web Font: `font-display: swap` + fallback フォントのメトリクス調整
- ヒーローセクションの最小高さを CSS で確保

#### Critical CSS
- ファーストビュー（ヘッダー + ヒーロー）の CSS を `<style>` でインライン化し、FCP を短縮

### 4-8. ブランドアイデンティティ

#### サイト名「k-ba-man」の扱い
- ロゴタイプとして扱い、統一された書体・ウェイトで固定
- ヘッダーでは「ロゴマーク + ロゴタイプ」のロックアップ
- ロゴマークは「K」を使った円形バッジ（最小 28px）。ファビコンにも流用
- サブタイトルはロゴタイプの下に小さく添える

#### 記号体系

| 記号 | 意味 | 表示ルール |
|---|---|---|
| ◎○▲△ | 印（本命/対抗/単穴/連下） | 常にこの4種。色分けはせず、テキストとして表示 |
| 席番号 01〜10 | キャラクターの固定識別子 | 円形バッジ（キャラカラー背景 + 白文字） |
| ランク数字 | 順位 | 等幅・大きめ・gold系カラー |
| ステータスピル | 的中結果（win/place/out） | pill 型バッジ。3色固定 |

#### 10色ストライプ
- 10人のカラーを順番に並べた帯（ゲートストライプ）はブランド要素
- ヒーロー下部、フッター上部等に1箇所のみ使用

## 5. ページ別デザイン要件

### 5-1. Overview

**目的**: 初見で世界観を掴み、リピーターは今週の入口へ直行する

- ヒーロー: プロジェクトのコンセプトを視覚的に伝える大型ビジュアル領域
  - キャッチコピー + サブテキスト
  - 主要アクションへの CTA（予想ライブ / 成績 / キャラクター）
- **予想ライブバナー**: 今週のレース名 + フェーズ + 直リンク（ヒーロー直下）
- **「今週のドラマ」帯**: 直近レース結果の物語的要約
- サマリー: 4つの主要統計をコンパクトに表示
- フォーカスセクション: 現在の首位キャラを紹介 + 上位3名のミニランク
- キャラクター一覧: 10人全員への導線
- **最新レース結果ダイジェスト**: 直近1レースの結果
- **ヒーローのビジュアル切り替え**: 直近の状況（首位キャラ、フェーズ）に応じて強調要素が変わる
- About セクション（折りたたみ）

### 5-2. 予想ライブ

**目的**: 今週のレース予想をリアルタイムに追う

- ページヒーロー: 対象レースの情報 + 現在のフェーズ表示
- フェーズナビゲーション: 3段階（告知→予想→結果）の可視化
- 集合知パネル: 10人の集約予想（印 + 確率分布）
- 個別予想カード: 各キャラの印・根拠テキスト・自信度
- **対立マーカー**: 集合知とずれた予想の視覚的強調
- 結果表示: 着順、収支、各キャラの答え合わせ、ランク変動
- 結果フェーズ: 各キャラの**直近N戦の◎結果**をインライン表示

#### モバイル戦略
- 620px 以下: フェーズナビを**セグメンテッドコントロール**（ピルボタン3つ横並び）に切り替え
- **個別予想カードの折りたたみ**: デフォルト閉じ、キャラ名・◎の馬名・自信度のみ表示。タップで展開

### 5-3. 成績

**目的**: 10人の序列とレースごとの成績を2つの切り口で見る

- ページヒーロー: 現在のレース数と注意事項
- **タブ切り替え**: ランキング表示 / レース別表示
- **ランキングタブ**:
  - ソートコントロール: 5種のソート基準を切り替え
  - ランキングテーブル: 順位 + キャラ情報 + 指標5列
  - **降格圏の視覚表現**: 暫定スコアの低い者への演出的ヒント
  - スコア解説
- **レース別タブ**:
  - 累計統計: 投資・払戻・収支・的中率
  - レースカード×N: 各レースの結果と収支

#### モバイル戦略
- 620px 以下: 各行は「順位 + キャラ画像・名前 + **ソート中の指標1つだけ**」を表示
- タップで詳細展開 or キャラ詳細へ遷移
- ソートラベルを短縮（「暫定席次」→「席次」、「本命実績」→「◎実績」）

### 5-4. キャラクター一覧

**目的**: 10人を一覧し、気になるキャラの詳細へ進む

- ページヒーロー: キャラクター紹介の導入
- カードグリッド: 10人分
  - 立ち絵を大きく
  - 名前・二つ名・成績サマリー
  - ホバーで誘導感

### 5-5. キャラクター詳細

**目的**: 1人のキャラクターを深く知る

- プロフィールヒーロー: 大型立ち絵 + キャラカラー背景
  - 名前、二つ名、流派
  - 座右の銘（引用）
  - 成績サマリー（4指標）
  - 前後キャラへのナビゲーション（**席番号順**。世界観上の固定識別子であり、成績で変動しない）
- **「この席に残る理由」**: drama テキストの独立視覚ブロック
- プロフィール本文: 紹介
- 強み・弱点パネル
- **関連キャラクター**: 対立・補完ペアへのリンク
- 成績履歴: レースごとのアコーディオン

#### モバイル戦略
- 620px 以下: **立ち絵を先、テキストを後**（ビジュアルファーストで「誰のページか」を即座に伝える）
- 立ち絵の高さは `min(50vh, 360px)` に制限し、ファーストビューにテキスト冒頭が入るように
- 前後ナビリンクを**ページ最下部にも複製**（thumb zone 配慮）

### 5-6. レース詳細モーダル

- 印、買い目一覧、払戻一覧、**勝ち筋のナラティブ**

#### モバイル戦略
- 620px 以下: **ボトムシート**として表示
  - 画面下端から出現、max-height: 92vh
  - 上部にドラッグハンドル、閉じるボタンは sticky
  - `border-radius` を上部のみに適用
- 背面スクロールは `body { overflow: hidden }` で抑止

### 5-7. 週次ドラマ（更新性の設計）

- **Overview ヒーロー下の「今週の円卓」帯**: レース結果データから算出するナラティブ
  - 席次変動の最大上昇/下降キャラ
  - 集合知◎の的中/外れ
  - 単独的中した少数派キャラ
- **鮮度マーカー**: ライブページや成績履歴に最終更新日時の控えめな表示
- **ヒーローの動的要素**: 首位キャラやフェーズに応じて強調が変わる

## 8. 物語的コンテンツの表示設計

### 8-1. 設計原則

サイト上の物語要素は、JSON データから算出可能な事実に基づく。人間が毎週テキストを書くのではなく、データの変化が物語を生成する構造。

### 8-2. 自動生成可能なナラティブ要素

| 要素 | データソース | 表示場所 |
|---|---|---|
| 席次変動のコメンタリー | 前回⇔今回の暫定スコア差 | 成績ランキング、ライブ結果 |
| 集合知vs個人の乖離 | collective.honmei ≠ expert.honmei | ライブ予想フェーズ |
| 「勝ち筋」ナラティブ | betting.how_won（既存フィールド） | レース詳細モーダル |
| 直近N戦の◎結果 | honmeiStatus の時系列 | ライブ結果、キャラ詳細 |

### 8-3. 関係性の可視化レベル

- **最小（初期）**: 対立ペアの名前を相互リンクで結ぶ（キャラ詳細の「関連キャラ」）
- **中間**: 対立軸ラベルを付けたペアカード（「物語 vs 検証」「穴 vs 守備」）
- **最大（将来）**: 円卓相関図ページ。初期スコープ外でも、将来追加可能な情報設計にしておく

### 8-4. 「勝ち筋」「答え合わせ」の扱い

- 「勝ち筋」（how_won）は数値の要約ではなく、物語的なヘッドラインとして設計
- 「答え合わせ」は「的中/外れ」の二値ではなく「何を見て、何が起きたか」を対比する構造
- 結果フェーズの各キャラカードに、honmeiStatus に応じた物語的文脈を添える余地を持つ

## 9. 成功基準

1. 初見ユーザーが5秒で「キャラクターが競馬を予想するメディア」と理解できる
2. キャラクター一覧で「この人の予想を見たい」と思えるビジュアルインパクトがある
3. 予想ライブページで「次のレースが楽しみ」と感じる情報設計になっている
4. データを見たい人が、指標の意味と各キャラの成績を正確に読み取れる
5. スマートフォンでの体験が劣化しない
6. リピーターが再訪時に**2タップ以内**で「今週の予想ライブ」に到達できる
7. 3レース以上のデータが蓄積した時点で、ランキング変動が「次回も見たい」というフックになっている
8. キャラクター詳細で「この人が外したときに何が起きるか」が伝わる
9. 予想ライブの結果フェーズが「答え合わせ」として読み物になっている
10. SNSで共有されたとき、OGP 画像とタイトルだけで「キャラ×競馬×AI」の世界観が伝わる
11. 10人を並べたとき、色・ポーズ・流派の違いが一瞬で読み取れ、「全員違う人」だと分かる

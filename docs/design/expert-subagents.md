## 1. コンセプトと理論的根拠

### 1.1 仮説

> **「日本のファーストネームを持つ10人の競馬予想専門家」を、思想（流派）・モデル（バックエンド）・思考深度（effort）で意図的に散らして並列に走らせ、集合知で集約すれば、市場オッズより高い精度で着順を当てられる。**

ハーネスの全戦略はこの一文に従う。10人の subagent は **独立に**（互いの予想を見せず）動き、後段の集約 skill が確率分布を統合する。

### 1.2 多様性予測定理（Scott Page）

集団予測の二乗誤差 **E** は、各個人の平均二乗誤差 **M** から「予測の多様性」**D** を引いたもの:

```
E = M − D
```

**含意**: D（多様性）を増やせば、M（個人の精度）が一定でも集団誤差は下がる。ただし条件付き — M を犠牲にして D を増やすと逆効果になる。**「精度の高い多様性」**（一定水準以上の精度を保ったまま、予測の仕方を散らす）が最適解。

### 1.3 散らし方の3軸

k-ba-man は次の3軸を **直交させて** 多様性を作る:

| 軸 | バリエーション | 狙い |
|---|---|---|
| **流派** | 血統 / データ統計 / 展開 / スピード指数 / 調教 / 騎手厩舎 / オッズ / 馬場 / 穴党 / 本命 | 思想（何を信じるか）の多様性 |
| **バックエンド** | Claude（Opus/Sonnet/Haiku）/ codex / gemini | モデル提供者・訓練データの多様性 |
| **effort** | 高 / 中 / 低 | 思考深度の多様性（即断と熟考の混在） |

10人の編成は 3軸を意図的に重複なく振り分ける（後述・§5 の多様性マトリクス）。

### 1.4 Surowiecki の 4 条件への対応

| 条件 | k-ba-man での実装 |
|---|---|
| 多様性 | 3軸の散らし（§1.3） |
| 独立性 | オーケストレーターが他者の予想を見せない（§3.3） |
| 分散性 | 流派が局所知識（血統DB、調教情報、オッズなど）に特化 |
| 集約 | 後段の集約 skill（log pooling + extremizing を採用予定。§7） |

## 3. 共通仕様

### 3.1 入力 — RaceDataPack

オーケストレーターが web スクレイピング・JRA 公式・netkeiba 等から **一括取得し、10人に同一を配布**する。スキーマ案:

```jsonc
{
  "pack_meta": {
    "version": "v1",                   // 同一レースで再取得したら v2, v3…（§3.5）
    "retrieved_at": "2026-06-13T21:00:00+09:00"
  },
  "race": {
    "race_id": "202609030811",         // netkeiba形式: 西暦4+場2(阪神=09)+回2+日目2+R2。開催回・日目は実装時に正式確定
    "name": "第67回宝塚記念",
    "grade": "G1",
    "date": "2026-06-14",
    "course": { "track": "阪神", "surface": "芝", "distance": 2200, "direction": "右", "inner_outer": "内" },
    "weather": { "forecast": "曇のち雨", "track_condition_forecast": "稍重" },
    "fan_vote": [ /* 上位馬の投票数 */ ]
  },
  "horses": [
    {
      "horse_num": 1, "horse_id": "...", "name": "クロワデュノール",
      "age": 4, "sex": "牡", "weight": 470, "weight_diff": -2,
      "jockey": { "name": "...", "id": "..." },
      "trainer": { "name": "...", "stable": "栗東" },
      "pedigree": { "sire": "...", "dam": "...", "broodmare_sire": "..." },
      "past_races": [ /* 直近 10 走（着順・走破時計・上がり3F・馬場・距離・コース） */ ],
      "training": [ /* 最終追切・週中追切 */ ],
      "odds": { "win": null, "place": null }    // 確定オッズが取得済みなら入る
    }
    // ...全頭分
  ],
  "track_bias": {
    "this_week_observations": "/* 当週の阪神芝の内外バイアス・前残り傾向など */"
  }
}
```

**設計判断**:
- 全員に**同じ**生データを渡す。「血統派は血統データだけ、調教派は調教データだけ」のように削るのは **しない** — 削ると独立した能力差が消える（同じ入力に対して、何を見るかの違いが多様性の源）
- 配布は **ファイルパス渡し**: オーケストレーターが `runs/<race_id>/pack-v1.json` に書き、10人にはパスだけ渡す。①全員がバイト単位で同一データを見ることを保証、②巨大 JSON を10回プロンプトに埋め込まない（トークン節約）、③CLI ラッパーは `cat` で読むだけ
- 各専門家は流派に応じて自分で web を追加で叩いてよい（gemini/codex は内蔵ツール、Claude 系は WebFetch / WebSearch）。ただしオーケストレーターの取得分は共通で、追加調査は **事実データに限る**（§3.3）

### 3.2 出力 — ExpertPrediction（10人共通スキーマ）

**これは将来の集約 skill が読む契約**。全員このスキーマで返す。

```jsonc
{
  "expert_id": "tatsunosuke",                    // 安定識別子（採点DB・runs/ のキー。jinba-<expert_id> と一致）
  "expert_name": "龍之介",
  "school": "血統",                              // 流派ラベル
  "backend": "claude-opus",                      // 採点分析用メタ（例: codex-gpt-5.5, gemini-2.5-flash）
  "effort": "high",                              // 設計軸の3段階: high / medium / low
  "race_id": "202609030811",                     // どのレースへの予想か
  "pack_version": "v1",                          // どの pack 世代を見たか（§3.5）
  "predicted_ranking": [                         // 着順予想（最大18頭、全頭でなくてよい）
    { "horse_num": 5, "rank": 1 },
    { "horse_num": 11, "rank": 2 },
    { "horse_num": 3, "rank": 3 },
    { "horse_num": 8, "rank": 4 },
    { "horse_num": 14, "rank": 5 }
    // 5着以下は省略可。集約 skill は省略を「同点」として扱う
  ],
  "win_prob": [                                  // 単勝確率分布（上位のみ列挙可）
    { "horse_num": 5, "prob": 0.32 },
    { "horse_num": 11, "prob": 0.18 },
    { "horse_num": 3, "prob": 0.12 }
    // 未列挙馬: 集約 skill が (1 − 列挙合計) / 残頭数 を割当（負なら 0 に切上げ）、
    // その後 全馬に ε=0.005 のフロア → 再正規化 を必ず適用（log pooling の veto 防止。§7.1）
  ],
  "place_prob": [                                // 任意: 3着内（複勝圏）確率。複勝で考える流派（優子等）は直接出す
    { "horse_num": 5, "prob": 0.65 }             // 省略時は集約 skill が win_prob から Harville 近似で導出
  ],
  "marks": {                                     // 印（伝統的可視化用、人間レビューも兼ねる）
    "honmei": 5,                                 // ◎
    "taikou": 11,                                // ○
    "tanana": 3,                                 // ▲
    "renka": [8, 14]                             // △
  },
  "confidence": 0.7,                             // 自己評価（0-1）。集約時の重み補正に使う
  "rationale": "クロワデュノール父系のスタミナ血統（…）。配合上、宝塚2200m内回りに適性極めて高い…",
  "data_used": ["pedigree", "past_races"],       // どの入力を主に使ったかのタグ
  "warnings": []                                 // 例: ["天気予報が変動中"]、空配列でも可
}
```

**重要な設計判断**:
- `win_prob` を必須にしているのは、後段で **probability pooling**（特に log pooling）を使うため。順位だけでは集約手法が Borda 系に縛られる
- `confidence` を分けて持つ理由: 同じ確率分布でも「自信度」が違うときに、log pooling の指数として使える（`P_agg ∝ Π P_i^{confidence_i}`）
- `rationale` は短く（200字程度）。人間レビュー用、集約には使わない
- ε フロアの責務は集約 skill 側に置く: 専門家には「上位だけ列挙してよい」軽い契約を保つ。フロア無しの log pooling は、陽菜が消した人気馬を集約からも消してしまう（veto effect）
- `place_prob` は任意。win_prob からの Harville 変換は本命-大穴バイアスで歪むことが知られているため、複勝圏で直接考える専門家の分は直接集約を優先する

### 3.3 独立性の原則

**他の専門家の予想は絶対に見せない**。これが集合知の生命線。

- オーケストレーターは10人の subagent を **並列起動**し、互いの中間出力にもアクセス不能にする
- 共有データは §3.1 の RaceDataPack だけ。10人が同じ pack を見るのは OK（同じ事実を見て独立に解釈するため）
- **オッズは herding 源**: オッズ・人気順をどう扱うかは流派ごとに方針を持つ。オッズ派（さくら）はオッズを主な信号とする、本命派（優子）は人気を信用する、穴党（陽菜）は意図的に逆張りする — 全員が同じ態度を取らないことが重要
- **追加 web 調査は事実データのみ**: 成績・調教時計・天気・馬場情報は可。**人間の予想・印・予想コラムを読むのは禁止** — 互いの予想を見せなくても、全員が同じ世論を読めば裏口から herding が起きる。例外はさくら（市場の動き自体が彼女の分析対象。ただしオッズ・投票データに限る — 予想家の印は不可）
- リポジトリ内の `data/experts/`（人間予想家の印・予想を収集したデータ）は**評価専用**（§7.3 のベースライン比較）。10人の専門家にも RaceDataPack にも**決して入れない** — 上記禁止事項のリポジトリ内版。さくらも例外ではない
- 集約 skill が走るのは10人が全員 ExpertPrediction を出した後。途中で「他の人は誰を本命にしたか」を訊いてはいけない

### 3.4 運用制約

| 項目 | 規約 |
|---|---|
| タイムアウト | effort 別: 低 120 秒 / 中 300 秒 / 高 600 秒（web 調査込みの CLI 高 effort 組に配慮。レースは週1なのでレイテンシは安い）。超過者は集約から除外。クォーラム: **8人以上**揃えば集約実行、7人以下なら Sho に相談 |
| 外部アクセス | 読み取り（GET）のみ。POST/PUT/DELETE 禁止（グローバルセキュリティルール） |
| 馬券購入 | **しない**。資金配分 skill の出力までで終わり（実購入は人間の Sho の判断） |
| API キー | 環境変数からスクリプト内で参照。Claude のプロンプトに直接展開しない（グローバルセキュリティルール） |
| CLI 不在時の fallback | gemini/codex バイナリが無い環境では、その専門家をスキップしてログに残す。集約は残ったメンバーで実行 |
| 確率の整合性 | 列挙した `win_prob` の合計は 0.95 〜 1.05 の範囲で許容。集約 skill が ε フロア（§3.2）付きで正規化 |
| 言語 | `rationale` 等の自由記述は日本語（kochi の母語ルール準拠）。プロンプト指示も日本語 |
| 実行ログ | pack と全 ExpertPrediction を `runs/<race_id>/` に保存（§3.5）。事後採点と w_i 更新（§7.1）の元データ |

### 3.5 実行タイミングと再実行

専門家を **「当日情報への感応度」** で3カテゴリに分け、再実行の要否とタイミングを変える。分類基準は **コア、または主要補正の入力が当日でしか確定しないか**。

| カテゴリ | メンバー | 当日入力 | 実行タイミング | pack |
|---|---|---|---|---|
| **静的組** | 龍之介・誠・葵・鉄平 | なし（入力が前日までに確定） | T-1日 夜のみ | pack v1 |
| **オッズ依存組** | さくら・優子・陽菜 | 当日オッズ・投票動向 | 当日朝（オッズが意味を持ち始める） | pack v2-odds（オッズ更新） |
| **馬場依存組** | 吾郎・美咲・健太 | 馬場差・含水率・クッション値・当週/当日の内外バイアス・天気実況 | **発走3時間前**（馬場が固まり、午前の同条件レースでバイアスが読める頃） | pack v2-baba（馬場系のみ更新・**オッズ非開示**） |

- **分類根拠**: 吾郎＝馬場がコア。美咲＝展開（前残り/差しの利きが馬場で反転する）。健太＝スピード指数が馬場補正を受ける（「吾郎の一言で表に補正線が入る」関係）。── この3人は当日馬場で評価が動く。さくら＝市場がコア、優子＝人気上位重視、陽菜＝人気の盲点で、いずれも当日オッズに感応。静的組（血統/統計/騎手厩舎/調教）は入力が前日までに確定する（道悪適性・当日気配などは「想定」として T-1 時点で織り込む）。
- **リーク防止（馬場依存組）**: 吾郎・美咲・健太はオッズ派ではない。`pack v2-baba` は **馬場・天気・バイアスだけを更新し、当日オッズや他者予想は載せない**（§3.3 の独立性・herding 防止。`build-betting-input` のホワイトリスト思想と同じ）。オッズ更新は `pack v2-odds` 側に閉じ込め、両者を混ぜない。
- **コスト整合**: 馬場依存組は3人とも Claude 直結（吾郎=opus-4-6/max、美咲・健太=Claude）。発走3時間前の再実行で codex/gemini のクォータに触れない。当日トリガーの単独再実行と相性が良い（オッズ依存組は優子・陽菜が gemini-flash を使う点が異なる）。
- **集約タイミング**: 当日依存組がいるため **最終集約は発走3時間前（馬場依存組の完了後）**。`pack_version` で世代を区別し、世代混在（静的=v1・オッズ=v2-odds・馬場=v2-baba）をその旨として集約結果に明記する。集約より下流（betting-input → 配分 → 買い目圧縮 → 提出）は発走までの残り時間で回す。
- **冪等性/フォールバック**: T-1日 夜の `pack v1` では当日依存組も **暫定値**を出しておく（前日オッズ・想定馬場で仮実行）。当日に本実行で上書きするが、当日トリガーが落ちても暫定値が残るので集約が破綻しない。

| 時点 | イベント | 動き |
|---|---|---|
| T-3日 | 枠順確定（発表済を確認） | pack 作成が可能になる |
| T-1日 夜 | 調教・天気予報・前日オッズが揃う | **pack v1 → 10人の基本実行**（静的組は本実行／当日依存組は暫定） |
| 当日朝 | 当日オッズが意味を持ち始める | `pack v2-odds` → **オッズ依存組（さくら・優子・陽菜）再実行** |
| **発走3時間前** | 馬場差・含水率・当日バイアスが固まる | `pack v2-baba` → **馬場依存組（吾郎・美咲・健太）再実行 → 最終集約** |
| T-50分 | 馬体重発表 | 原則追わない（全員再実行のコストに見合わない） |

- 全実行の pack・ExpertPrediction は `runs/<race_id>/` に保存:

```
runs/<race_id>/
├── pack-v1.json
├── pack-v2-odds.json            # 当日オッズ更新（オッズ依存組用）
├── pack-v2-baba.json            # 馬場系のみ更新・オッズ非開示（馬場依存組用）
└── predictions/
    ├── v1/<10人分>.json          # 基本実行（当日依存組は暫定）
    ├── v2-odds/{sakura,yuko,hina}.json
    └── v2-baba/{goro,misaki,kenta}.json
```

## 4. 10人のプロフィール

各人物の表記:
- **人物像**: 性格・口調（rationale に表れる色付け。集合知の同質化を防ぐため、個性を意図的に持たせる）
- **流派**: §1.3 の流派軸
- **バックエンド**: §1.3 のモデル軸（実呼出方法）
- **effort**: §1.3 の思考深度軸
- **宝塚記念2200mでの着眼点**: 目標レース固有の見方
- **データ要求**: §3.1 の RaceDataPack のどこを優先して見るか
- **プロンプト骨子**: 各 `.claude/agents/<name>.md` のシステムプロンプトの核（全文化は実装フェーズ）

- **データ要求**: `horses[].past_races[].time` から指数を推定、または外部 API（JRDB 等）から取得
- **プロンプト骨子**:

| 高 | 血統（龍之介）/ 統計（誠）/ 馬場（吾郎） |
| 中 | 展開（美咲）/ 調教（鉄平）/ 騎手（葵）/ オッズ（さくら） |
| 低 | 指数（健太）/ 穴党（陽菜）/ 本命（優子） |

→ 高 3 / 中 4 / 低 3 のバランス。**「即断即決もプロ」** として低 effort 組を構成（高 effort 偏重を避け、思考深度の多様性を確保）。

### 5.3 バックエンド × effort

| | 高 | 中 | 低 |
|---|---|---|---|
| Opus | 龍之介・吾郎 | 美咲 | |
| Sonnet | | 葵 | 健太 |
| Haiku | | さくら | |
| codex | 誠 | 鉄平 | |
| gemini | | | 陽菜・優子 |

→ 当初は Opus 2 / Sonnet 2 / Haiku 2 / codex 2 / gemini 2 で均等・**全セル1人以下**（完全直交）だった。**2026-06-20 に吾郎を gemini-2.5-pro→Claude opus-4-6/max、優子を Haiku→gemini-2.5-flash へ移行**（gemini 無料枠の安定運用の都合）。これにより Opus×高（龍之介・吾郎）と gemini×低（陽菜・優子）が各2人となり、完全直交は崩れた。effort 配分 3:4:3 と流派の独立性は維持。

### 5.4 多様性スコア（定性チェック）

- ✅ 同じ流派が 2 人いない
- ⚠ バックエンド × effort は当初全セル1人以下だったが、2026-06-20 の移行で Opus×高（龍之介・吾郎）と gemini×低（陽菜・優子）が各2人に。流派の独立性は保つが完全直交ではない（§5.3）
- ✅ 「人気馬を信頼する側」（優子・健太・誠）と「人気馬を疑う側」（陽菜・さくら・吾郎）の対抗構造
- ✅ 「血統・展開・調教」のような **データに表れにくい流派** と「指数・統計・オッズ」のような **数値主導流派** の両方を含む
- ⚠ プロバイダ単位では **Anthropic 6 : OpenAI 2 : Google 2** — 同一プロバイダのモデルは訓練データ・癖が相関するため、実効多様性はモデル数より小さい。ホーム（Claude Code）の安定性を優先した意図的な偏り。運用で Claude 組の予想相関が高いと判明したら 4:3:3 への再配分を検討

## 7. 将来スコープとの接続

### 7.1 集約 skill（次フェーズ）

10 人の `ExpertPrediction[]` を受け取り、`AggregatedPrediction` を返す skill。設計書としてのスコープ外だが、本設計書の `ExpertPrediction` スキーマがそのまま入力になる契約。

採用予定の手法（リサーチ [collective-intelligence.md](../research/collective-intelligence.md) より）:

1. **Logarithmic Pooling** を主、Linear Pooling を補助:
   ```
   P_agg(着順) ∝ Π_i P_i(着順)^{w_i}
   ```
   `w_i` は `confidence` × 過去精度の関数

2. **Extremizing**: pooling 後の確率を尖鋭化（herding で平坦化されたぶんを復元）。ただし log pooling 自体に尖鋭化作用があるため、係数は控えめな初期値から実レースで調整する

3. **重み w_i の決定**:
   - 初期は等重み（過去データなし）
   - 運用後は rolling window 20 レースの Brier score / Log Loss から動的更新
   - **較正は実レース（knowledge cutoff 後）の結果のみで行う** — 過去レースのバックテストは §7.3 の汚染問題があるため w_i の根拠にしない

4. **順位集約**: 上位 8 頭に対しては Kemeny-Young 近似で精密化（馬連・三連複向け）

**重みづけ運用方針（2026-06-14 追記・実運用の合意事項）**

w_i の導入は「**集合知を弱めない範囲で**」行う。多様性予測定理 `E = M − D` より、重みづけは効果的な個人精度 M を上げる一方で多様性 D を下げうる（声が似てくる）。差し引きが正になる保証はない。よって運用ルール:

1. **等重みが既定**。重みは「過去精度で重みづけた集約が、等重み集約を *新しいレースで（out-of-sample）* 上回る」と実証できたときだけ ON にする。金融の 1/N 効果（重み推定のノイズが利得を食う）と同じく、等重みは強いベースライン。
2. **報酬は当たり外れ(hit)ではなく較正(Brier/LogLoss)＋集団への限界貢献**。「外したが独立した角度で見ている」専門家は D を支えるので、単独成績が平凡でも安易に下げない。理想は「その専門家を抜くと集団誤差がどれだけ悪化するか」で測る。
3. **総取り禁止・緩い傾斜**。重み比に上限を設け（例: 最大でも数倍）、全員をプールに残す。少数派でも較正がよく独立なら生かす。2026宝塚で3着内を獲ったのは少数派（血統・逆張り）の独立票だった — それを数レースの不調で消すのが最悪手。
4. **n 規律**。ウォームアップ期間（`data/scoring/weights.json` の `min_races_before_weighting`＝20レース）まで等重きを維持し、スコアは `per_race` に蓄積するのみ。到達後は直近20レースを rolling window として w_i を算出する。較正は knowledge cutoff 後の実レースのみ（§7.3）。
5. 現状 `aggregate.mjs` は自己申告 `confidence` で重みづけするが、レビューで confidence は狭帯域に密集しほぼ無情報と判明。上記の過去精度はこの無情報な自己申告を**置換／補正**する位置づけ。

採点・台帳の実装は `scripts/score-race.mjs` と `data/scoring/`、週次運用は [weekly-ops.md](./weekly-ops.md) を参照。

### 7.2 資金配分 skill（次々フェーズ）

`AggregatedPrediction` + 現在オッズ → 各馬券（単勝・複勝・馬連・三連複）の購入額。

採用予定: **fractional Kelly criterion**:
```
f* = (b * p − q) / b      # 全 Kelly
bet = bankroll * f* * k   # k = 0.25 程度の保守版
```
- `b`: オッズ - 1（純利益倍率）
- `p`: 集約予測の的中確率
- `q`: 1 - p
- 期待値 `b * p > q` のときのみ賭ける

**この設計書の `win_prob` がそのまま `p` の入力**。複勝・馬連等は確率分布から導出（`place_prob` を出した専門家の分は直接集約を優先）。

注意: 複数券種の同時購入は結果が相関するため、券種ごとの独立 Kelly の単純合算は過大賭けになる。同時 Kelly（joint Kelly）か総額上限で抑える — 資金配分 skill 設計時の必須要件。

### 7.3 バックテストと汚染問題

過去レースで10人を仮想的に走らせる構想には**根本的な罠**がある: LLM は訓練データに過去レースの結果を含むため、「2024年の宝塚記念を予想して」は予想力ではなく**記憶力の測定**になる（成績が体系的に上振れる）。

- 過去レースでのバックテストは**性能の上限値の参考**としてのみ扱い、w_i の較正・手法選択の根拠にしない
- 真の検証は knowledge cutoff 後の実レースのみ: 2026 宝塚記念を初回として、以後の重賞で逐次評価（Brier / Log Loss / RPS / 回収率を `runs/` から採点）
- レース後採点は専用の採点 skill として実装する（§8.5）

レース後採点では、集約予想を **3つのベースライン** と比較する:

1. **市場ベースライン** — 確定オッズ由来の暗黙確率（控除率補正後）。これに勝てなければ集合知をやる意味がない
2. **人間予想家ベースライン** — `data/experts/<race>.json` に収集した予想家の印・消し根拠（宝塚記念2026 分は収集済み: うましる等、2026-06-12 取得）。印（◎○▲△）は predicted_ranking 相当に変換して同じ指標で採点。**AI 集合知 vs 人間の専門家**という本プロジェクトで一番面白い比較軸
3. **個人 vs 集団** — 10人それぞれの単独成績 vs 集約結果。多様性予測定理（E = M − D）が実データで成立しているかの直接検証

## 8. セルフレビュー結果

### 8.1 出力スキーマが集約計算にそのまま使えるか

✅ `win_prob` は probability pooling（log / linear）にそのまま投入可能。  
✅ `predicted_ranking` は Borda / Kemeny に投入可能（5 着以下省略を「同点最下位」として扱う規約あり）。

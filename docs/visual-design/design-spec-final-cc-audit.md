## A. critical 所見（実証済み・統合後6件）

### A1. 【最重要】サーフェス OKLCH 値が現行 hex の実値と全段で乖離（technical + migration が独立に確認）

- **主張**: §2-2 / §1 R1 / 付録A は `--surface-0..3 = 0.11/0.14/0.18/0.22` を「現行 `#0b0c11/#111319/#181a24/#20222e` の**忠実な oklch 移行**」とし、「現行 `#0b0c11` の実 oklch L は ≈0.11」と断言。
- **問題**: これは誤り。標準 Oklab 変換（`#000=0 / #fff=1 / #808080=0.6` で検証済み）の実測は **`#0b0c11`=L0.156 / `#111319`=L0.187 / `#181a24`=L0.220 / `#20222e`=L0.255**。本書の各値は対応する現行色より **L で 0.035〜0.047 暗く**、彩度も実測 0.011〜0.023 に対し一律 0.01 と低い。採用すると**サイト全体が現行より明確に暗くなる**。
- **派生**: R1 が「0.08 案は現行≒0.11 と矛盾」として却下した根拠の数値自体が虚偽（実値0.156。なお0.08は0.156からさらに遠いので「0.08不採用」の結論は維持可）。`--border 0.16 ≈ #1c1e2c` / `--border-mid 0.20 ≈ #252738` も同根で誤り（実値 L0.240 / L0.279）。上流 `design-spec-re-plan.md:34` の `#0b0c11 ≒ oklch(0.11)` を検証せず継承したのが原因。
- **推奨**: 二択。**(a) 忠実移行を意図** → 実測値 `oklch(0.156 0.011 276)/(0.187 0.013 271)/(0.220 0.020 276)/(0.255 0.023 277)`（丸めるなら 0.16/0.19/0.22/0.26）に修正。border も `0.24/0.028` に。**(b) 意図的に暗くする** → 「忠実移行」表記を撤回し設計判断として明記。いずれにせよ「≈0.11」「忠実移行」の併記は不可。

### A2. 存在しないルート `#/results` を実装前提に複数箇所で参照（technical + coverage）

- **問題**: §3-3 タブ4項目「成績(#/results)」・§5-3・§14 が `#/results` を参照するが、実 app.js のルートは `{overview, ranking, live, characters, races}` の**5本**で `results` は存在しない（現行ナビも5項目）。`#/results` は design-spec v2 が掲げる**統合後の目標**ルート名であり、final-cc はそれを「既存」かのように書き、5→4統合という最重要 IA 移行（`#/ranking`+`#/races` → `#/results`、router 改修必須）を明記していない。R2「JS構造変更なし」とも干渉。
- **推奨**: 「現行 `#/ranking` + `#/races` を `#/results` に統合（router 改修・旧ルートはリダイレクト）、ナビ5→4」という現行→最終マッピングを明記。タブ4項目のリンク先も整合させる。

### A3. 「§2 が唯一の供給源」の破れ — `--gold/--gold-light/--gold-glow/--red/--green` が §2 未定義（consistency）

- **問題**: §0/§2 は「§2 が全色・全寸法の唯一の供給源」と明言するが、§4 等の component CSS が `var(--gold)` `var(--gold-light)` `var(--gold-glow)` `var(--red)` `var(--green)` をフォールバック無しで参照している一方、これらは §2 のどの表・`:root` ブロックにも**定義されていない**（現行 styles.css の :root にのみ存在＝暗黙の継承依存）。
- **推奨**: §2 に gold/red/green 系を「継承トークン」として正式列挙するか、§2-4 の `--status-*` を実体トークンに昇格して参照させる。「唯一の供給源」の主張と実体を一致させる。

### A4. `--status-win/place/out/up/down` が定義のみで死蔵、実装は3系統バラバラ（consistency）

- **問題**: §2-4 は「成績色は全ページ統一」と宣言し `--status-*` を定義するが、本文での `var(--status-*)` 参照は**0件**。実コンポーネントは同一概念を **3通り**で着色: 例 win = §2-4表 `oklch(0.74 0.10 90)` / status-pill `var(--gold)` / form-dot `oklch(0.8 0.15 85)`。宣言と実装が矛盾。
- **推奨**: 1概念=1トークンに正規化。`--status-win/place/out` を実体トークンとして確定し、status-pill・form-dot・rank-number-danger から `var(--status-*)` で参照。現行 rgba/hex は「移行目標↔現行実装」対応表に留め、component にリテラルを散らさない。

### A5. `--accent-focus` を「全キャラ色と非干渉」と誤断言 — aoi と完全同色相（a11y・実計算）

- **問題**: §2-9 と §8 が `--accent-focus = oklch(0.75 0.15 250)` を「hue250・全キャラ色と非干渉」と2度断言。だが `--c-aoi = oklch(0.50 0.17 250)` で**色相が完全一致(250°)**。aoi 要素上でフォーカスリングが同色相になり識別性が落ちる。
- **推奨**: 「非干渉」を撤回し、focus 色相をキャラ未使用帯（例 hue 330 付近）へ移すか無彩色寄りに。最低限「aoi と同色相」である事実を明記。

### A6. 10色パレットの「色相差 ≥30°・sub-30°は4ペア」が誤り — 実際は6ペア（a11y・実計算）

- **問題**: §2-3 は「隣接色相差は原則≥30°、sub-30°近接は4ペアのみ」とするが、ソート色相環の実測では sub-30° は**6ペア**。本書が見落とした **teppei(85°)↔goro(110°)=25° が最小**、goro↔kenta=28° も該当。成功基準#11「全員違う色」の根拠が崩れる。
- **実測色相環**: sakura8→tatsunosuke28=**20°** / →hina52=**24°** / →teppei85=33 / →goro110=**25°(未記載)** / →kenta138=**28°** / →yuko172=34 / →makoto200=**28°** / →aoi250=50 / →misaki300=50 / →sakura=68。
- **推奨**: sub-30° を6ペアに訂正し各ペアの L/C 分離を明記。特に teppei↔goro は dL0.04・dC0.04 と近すぎるので goro の L を下げるか C を上げる。

## B. major 所見（16件・要点）

**色・トークン系**
- **B1 明度帯分離が自値で破綻**（§2-4）: `status-out oklch(0.55 …)` と `sakura(キャラ) oklch(0.55 …)` が同明度の赤。「~0.15 差で誤読しない」は不成立。形状分離(pill vs 円)を主軸に据え、明度差0.15の具体値クレームは撤回。
- **B2 `--muted` の AA「✓」が過剰**（§2-5/R5）: `#7a7c92` は surface-0 上 4.77:1（可）だが **surface-2 上 4.23:1・surface-3 上 3.85:1 で 4.5:1 未達**。doc 自身の付録B#7 が懸念しながら R5 表は「✓」断言。L≈0.58 へ引き上げるか、用途を surface-0/1 限定と明記。
- **B3 カードの focus-visible が accent-vivid で 3:1 未達**（§4-1）: tatsunosuke 2.40 / misaki 2.84 / goro 2.88（対 surface-1）。§8「非テキストUI 3:1」違反。リングは堅牢な `--accent-focus`（対全surface 7:1超）を使えば解決。
- **B4 席バッジ白文字 AA の失敗候補が不正確**（§4-2/付録B#5）: 実測では **hina のみ確定未達(4.18:1)→.seat-badge--light-bg必須**。makoto/kenta は4.59:1の境界、**teppei は白で5.54:1と安全**（doc は teppei を失敗候補と誤推測）。
- **B5 `--border/--border-mid` の oklch 等価値が誤り**（§2-2）: A1 と同根。実値 L0.24/L0.28、C も0.027〜0.031。

**整合・トークン運用**
- **B6 `--shadow-1/2/3`・`--ease-default` が定義のみ未参照**（§2-9）: 立ち絵は inline drop-shadow リテラル直書き。`--shadow-*` を「矩形要素専用」と用途明記しカード/モーダルから参照させるか削除。
- **B7 R8(dvh統一)が §5-5 で未適用**: キャラ詳細ヒーローに `70vh`/`50vh` の plain vh が残存（design-spec.md:428 をそのまま継承）。`70dvh`/`min(50dvh,360px)` に統一。
- **B8 `--header-h` が §2 正本に無く §5-1 散文で初出**: ヒーロー高さ計算が依存する基本寸法を §2-7 :root に正式定義する。

**ブラウザ互換**
- **B9 `@starting-style` 「Safari 未対応」は事実誤認**（§2-10）: Safari 17.5+(2024-05)で対応済み。真の制約は「動的DOM挿入要素に効かないバグ」。結論（開きのみ・閉じ即時）は妥当だが理由を正しい事実に差し替え。
- **B10 R10 フォントメトリクスの出典取り違え**（§2-6/R10）: 「typography=105/25/102 / performance=88/22/100」は誤り。実際は **typography研究=88/22/100、re-plan=105/25/102**。§2-6 は re-plan 値を載せつつ typography 由来と誤記。出典を正し、どちらを初期値にするか正直に明記（research準拠の 88/22/100 を推奨）。

**要件トレーサビリティ（design-spec v2 からの欠落）**
- **B11 暫定スコア計算式・席次タイブレークの欠落**: v2 §3-3 の `max(0,(1-meanBrier/2)*80)+…`（app.js:283-288 と一致）が final-cc に再掲も参照も無い。「これ1本で実装着手」を標榜する以上、式を1ブロック再掲か「正本は design-spec §3-3」と明示参照。
- **B12 ページ間導線マトリクス・「ハブ」原則の脱落**: v2 §2-3 の12行優先度マトリクスと「キャラ詳細=最多導線のハブ」が carried over されていない。要約か明示参照を追加。
- **B13 成功基準#4 の指標説明が hand-wave**: LogLoss/網羅率の「意味の説明」本文が final-cc に無い（§4-4 のソートkey名のみ）。§5-3「スコア解説」に5指標の1行定義を載せる要件を明記。
- **B14 ナビ5→4統合の現行→最終差分が未明記**: A2 と同根。最重要 IA 移行を1行で明記。
- **B15 relationships データ不在の未明示**: v2 が要求する構造化 `relationships` は現行データに皆無（app.js の weakness に名前埋め込みが現存＝v2禁止の旧方式）。付録B に「relationships 新設まで §11 関係性UIは描画不可」を P1 前提タスクとして追加。

**現行コード整合**
- **B16 live グローバル名の誤り**（§5-1）: `__LIVE_RACE__` は存在しない。実体は `window.KBAMAN_LIVE_RACE`（app.js 変数は `LIVE_EVENT`）。条件分岐の前提名を修正。

## D. 強み（calibration — 健全な点）

監査は「全部ダメ」ではない。特に**事実関係の正確性が高く、ハルシネーションがほぼ無い**:

- **リポジトリ主張がほぼ全件 VERIFIED**（migration観点）: 現行4 surface hex、`--muted #62647a`・`--ink`・`--text`・`--max 1180px`・border 系、**color-mix は実21箇所・全て in srgb・0 oklch**で引用行番号も正確、**10キャラ hex は app.js とゲートストライプにバイト一致・席順01-10通り**、`--accent` 注入8箇所・`--char-color` は0、index.html は defer無し5本/OGP・favicon無し/viewport-fit無し、cwebp導入済み・avifenc不在、画像サイズ実測一致。
- **テキストコントラスト主張は保守的に正しい**（実測 ink 16.23 / text 7.93 / muted-on-S0 4.77）。`--muted` の AA 修正は実バグ（現行 #62647a=3.37:1）の正しい是正。
- **ブラウザ互換クレームが概ね正確**: color-mix Baseline 2023、View Transitions Baseline 2025、RCS ~89.6%、oklch per-channel var() 不可、`0.01ms`の理由、`@media`に`var()`不可、lazy+fetchpriority併用禁止。
- a11y パターン（aria-sort アクティブ列のみ・属性削除、route-announcer 常設 polite、立ち絵 alt 3規則、色を唯一の信号にしない）は WCAG/APG 準拠。
- 降格圏 gating・立ち絵表示表（値まで）・週次ドラマのテンプレ分解・R2/R9 が JS を壊さない判断・暫定スコア式の読解は v2 に忠実。

## F. 注記: auditor 間で割れた1点（要追加検証）

- **10色パレットの sRGB ガマット**: 技術観点は「10色すべて sRGB ガマット内（sakura0.20/misaki0.18 もクリップせず）」とし、a11y観点は「makoto/teppei/aoi/hina/yuko の5色がガマット外でクリップされ実色がズレる」とした。**両者が矛盾**。パレット採用（R9・P1）の前に、最終10 oklch 値を sRGB ガマット判定し、クリップされる色があれば C を調整すること。
